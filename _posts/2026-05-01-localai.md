For the past several years I have been thinking carefully about what it means to run AI infrastructure that I actually own, control, and understand from the ground up. The rapid proliferation of frontier model APIs, agentic coding tools, and open-weight model releases in 2025-2026 finally made this tractable at a price and complexity point that a single person could manage. This post documents the architecture I settled on: a self-hosted, Docker-based stack running on a mini PC, unified by a single OpenAI-compatible model gateway, and surfaced through a collection of local inference servers, agentic CLI tools, autonomous agent frameworks, open-source Cowork alternatives, and a task-bounded command harness built around structured queues. My goals were privacy, sovereignty, reproducibility, and the ability to swap components without rebuilding everything from scratch.
 
## Motivation: Why Self-Host?
 
The short answer is control. When I work on grant-funded AI research, on student data in the context of my courses at Ursinus, or on institutional planning through my role as Co-Director of the Teaching and Learning Institute, I want to know exactly where inference is happening and what data is leaving my environment. The longer answer is pedagogical: I cannot credibly teach AI literacy, AI ethics, and responsible deployment if I have not made serious, hands-on architectural decisions myself. Running your own stack is humbling in the right ways.
 
There is also a strategic argument. As the open-source AI agent framework ecosystem matured through early 2026, it became clear that the layered architecture of these systems, separating the model layer, the orchestration layer, the tool-access layer, and the user-interface layer, was stabilizing into recognizable patterns. A well-designed self-hosted stack can plug into any of these layers without being locked to a single vendor. That flexibility is worth the setup cost.
 
## Hardware: The Mini PC
 
The physical foundation of the stack is a mini PC running Linux Mint, which gives me a clean Debian-lineage environment with full access to the upstream Docker Engine repository and no virtualization layer between the container workloads and the host kernel. Everything runs directly on the host, which simplifies networking, volume permissions, and service lifecycle management considerably compared to hypervisor-based setups.
 
### Installing Docker Engine
 
Linux Mint ships an older Docker build from the distribution mirror, so I install from the upstream Docker repository instead. The setup sequence is:
 
```bash
# Remove any distribution-packaged versions
sudo apt-get remove docker docker-engine docker.io containerd runc
 
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release
 
# Add Docker's official GPG key
sudo mkdir -m 0755 -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
 
# Point the apt source at the Ubuntu codename that Mint is based on
echo \
  "deb [arch=$(dpkg --print-architecture) \
  signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo $UBUNTU_CODENAME) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
 
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin
 
# Verify
sudo docker run hello-world
 
# Allow your user to run Docker without sudo
sudo usermod -aG docker $USER
newgrp docker
```
 
The `UBUNTU_CODENAME` variable in the source line is the key detail for Mint: it resolves to the underlying Ubuntu release name rather than the Mint release name, which is what the Docker repository actually indexes.
 
### Installing Ollama
 
Ollama runs as a native Linux service, not inside a container, which keeps model-weight I/O off the Docker networking path and avoids bind-mount overhead for the weight files:
 
```bash
curl -fsSL https://ollama.com/install.sh | sh
# The installer registers a systemd service; verify it started
systemctl status ollama
```
 
Ollama listens on `http://localhost:11434` by default and is reachable from within containers via `--add-host=host.docker.internal:host-gateway`, which every container in the stack declares. This flag is required on Linux Docker Engine; unlike Docker Desktop, the Linux engine does not inject `host.docker.internal` automatically.

The workspace is simply a directory in the home filesystem, accessible to Docker via ordinary bind mounts and to the host user with full POSIX permissions.
 
## The Core Design Principle: A Unified Model Gateway
 
The single most important architectural decision I made was to route all LLM inference through a single OpenAI-compatible endpoint rather than having each tool reach out to Ollama, Anthropic, or OpenRouter independently. I use [LiteLLM](https://github.com/BerriAI/litellm) for this. Every service in the stack sends its requests to `http://localhost:4000/v1` with the bearer token `sk-litellm-local`. LiteLLM translates those requests to whatever backend is appropriate, whether a local Ollama model, a LocalAI GGUF endpoint, or an OpenRouter free-tier model, according to a YAML routing configuration.
 
```yaml
model_list:
  - model_name: llama3
    litellm_params:
      model: ollama/llama3
      api_base: http://host.docker.internal:11434
 
  - model_name: qwen2.5-3b
    litellm_params:
      model: ollama/qwen2.5:3b
      api_base: http://host.docker.internal:11434
 
  - model_name: hermes3
    litellm_params:
      model: ollama/hermes3:8b
      api_base: http://host.docker.internal:11434
 
  - model_name: localai/phi4-mini
    litellm_params:
      model: openai/phi4-mini
      api_base: http://host.docker.internal:8080/v1
      api_key: ignored
 
  - model_name: openrouter/auto
    litellm_params:
      model: openrouter/auto
      api_key: ${OPENROUTER_API_KEY}
      api_base: https://openrouter.ai/api/v1
```
 
The practical consequence is that changing a model, adding a provider, or adjusting routing requires editing one YAML file and restarting one service. No other container needs to know that anything changed. This is the same composability principle I teach in software engineering: minimize coupling, maximize cohesion.
 
Connecting a commercial CLI tool like Claude Code to the local stack requires only two environment variable exports:
 
```bash
export ANTHROPIC_BASE_URL=http://localhost:4000
export ANTHROPIC_API_KEY=sk-litellm-local
```
 
From Claude Code's perspective, nothing has changed. From the infrastructure's perspective, every token has been processed locally.
 
## The Full Stack: Services and Ports
 
Running everything in Docker Compose or as standalone containers gives me clean lifecycle management, easy restarts, and clear separation between persistent identity state (credentials, config) and ephemeral compute. The full service map is as follows.
 
| Service | Port | Purpose |
|---------|------|---------|
| LiteLLM | 4000 | Unified model gateway (OpenAI-compatible) |
| Ollama | 11434 | Local LLM inference (native systemd service) |
| LocalAI | 8080 | GGUF model inference (OpenAI-compatible) |
| Open WebUI | 3000 | Browser-based LLM frontend with MCP tool calling |
| Mastra | 4111 | TypeScript AI agent server (API + Studio UI) |
| Agent Zero | 8081 | Autonomous hierarchical agent with web UI |
| Archon | 8501 | Meta-agent framework for building agent systems |
| Portainer | 9000 | Docker management UI |
 
The sections below describe each layer in operational detail.
 
## Local Inference: Ollama and LocalAI
 
The stack runs two local inference backends with distinct tradeoffs, and LiteLLM routes between them transparently based on model alias.
 
**Ollama** is the primary inference backend for the models I reach for most often. Its systemd service model keeps it available before Docker is fully up, its model management CLI (`ollama pull`, `ollama list`, `ollama rm`) is clean, and its HTTP API is stable. The models I maintain in Ollama are selected for RAM footprint first: `phi4-mini`, `smollm2`, `gemma4:e2b`, `qwen2.5:1.5b`, `qwen2.5:3b`, `llama3`, and `hermes3:8b`. All of these run comfortably in CPU-only mode without swapping, which matters because other services are competing for the same physical memory. The Hermes 3 checkpoint from Nous Research deserves special mention: it is one of the most reliably well-behaved models for function-calling workflows, and I route all Open WebUI tool-use sessions through it by default. Its instruction-following is consistent, its JSON output conforms to the function-calling schema without coaxing, and it handles multi-turn tool exchanges without losing track of prior tool results.
 
**LocalAI** ([github.com/mudler/LocalAI](https://github.com/mudler/LocalAI)) is the secondary inference backend, running on port 8080 behind an OpenAI-compatible API surface identical to Ollama's. The distinction that makes it worth running alongside Ollama is backend flexibility: LocalAI supports not just llama.cpp for text generation, but also whisper.cpp for speech transcription, stable diffusion for image generation, and bark for text-to-speech, all behind the same endpoint. A single `POST /v1/chat/completions` call routes to the appropriate backend based on the model name. LocalAI is organized around three directories that are bind-mounted into its container: `localai/models/` holds GGUF weight files, `localai/backends/` holds compiled backend binaries that LocalAI downloads on first use, and `localai/config/` holds per-model YAML configuration files that specify backend selection, context length, and GPU layer offload parameters.
 
```yaml
# localai/config/phi4-mini.yaml
name: phi4-mini
backend: llama
parameters:
  model: phi-4-mini-instruct.Q4_K_M.gguf
  context_size: 8192
  threads: 8
```
 
The practical role LocalAI plays in my stack is model experimentation: I use it to try GGUF checkpoints before deciding whether to promote them to the Ollama catalog, and for any workload that needs the non-text modalities that Ollama does not yet support natively.
 
## Browser Frontend: Open WebUI
 
[Open WebUI](https://github.com/open-webui/open-webui) is the browser-based interface for direct LLM interaction in my stack, running on port 3000. Its design is deliberately open: the interface is model-agnostic, the backend is self-hosted, and nothing requires a cloud account or API key if Ollama is the only backend. It connects to Ollama directly via `http://host.docker.internal:11434` and enumerates available models automatically, which means a newly pulled Ollama model appears in the model selector without any additional configuration step.
 
The feature that makes Open WebUI genuinely useful rather than merely functional is its native MCP tool-calling support, introduced in version 0.4. When a function-calling capable model produces a tool-call response, Open WebUI intercepts it, dispatches the call to the appropriate registered MCP server, and injects the result back into the conversation as a tool result message, all within a single chat turn. The models I use for tool-calling sessions are `hermes3:8b`, `llama3.1:8b`, `qwen2.5:7b`, `qwen2.5:14b`, and `mistral-nemo:12b`, all of which produce consistently well-formed tool-call JSON.
 
Open WebUI also maintains full conversation history in a SQLite database mounted from `openwebui/data/`, which survives container restarts and is accessible for export. The pipeline system, introduced in the same version, allows custom Python preprocessing steps to run between the user's input and the model, which I use for lightweight document injection and prompt augmentation in cases where I want context added without modifying the user-facing interface.
 
```bash
docker run -d \
  --name open-webui \
  --add-host=host.docker.internal:host-gateway \
  -p 3000:8080 \
  -v "$HOME/agents/openwebui/data:/app/backend/data" \
  -e OLLAMA_BASE_URL=http://host.docker.internal:11434 \
  ghcr.io/open-webui/open-webui:main
```
 
## Agentic CLI Tools: A Comparative Survey
 
By April 2026, a mature set of agentic coding CLI tools has emerged with distinct architectural philosophies, and I run all of them through the unified LiteLLM gateway. The comparative picture is useful to have in one place.
 
**Claude Code** (Anthropic, Node.js) is the most fully featured in terms of built-in subagent support, MCP integration, and permission gate granularity. It uses `CLAUDE.md` files for project context and `.claude/agents/` Markdown files for custom subagent definitions. Connecting it to the local stack requires only setting `ANTHROPIC_BASE_URL` and `ANTHROPIC_API_KEY`.
 
**OpenAI Codex CLI** (Rust) is the most efficient in terms of resource consumption and supports native multi-provider configuration through a TOML config file. Custom providers are defined as named sections:
 
```toml
model = "llama3.3:70b"
model_provider = "openwebui"
 
[model_providers.openwebui]
name = "Open WebUI"
base_url = "http://localhost:3000/openai"
env_key = "OPENWEBUI_API_KEY"
```
 
**Gemini CLI** (Google, Node.js) uses `GEMINI.md` files for project context and a three-tier discovery hierarchy for skills. Routing it through an OpenAI-compatible endpoint requires the `open-gemini-cli` fork, which injects an adapter layer that translates Gemini's internal message format.
 
**OpenCode** (opencode.ai, Go) is the most flexible in terms of provider support, relying on the `@ai-sdk/openai-compatible` adapter to connect to any OpenAI-compatible backend, including the local LiteLLM instance:
 
```json
{
  "provider": {
    "litellm": {
      "npm": "@ai-sdk/openai-compatible",
      "options": { "baseURL": "http://localhost:4000/v1" },
      "models": { "llama3": {}, "qwen2.5-3b": {}, "hermes3": {} }
    }
  }
}
```
 
**KiloCode** (Node.js, VS Code extension) is the VS Code-native member of the survey. It is an evolution of the Cline/Roo Code agentic coding extension lineage, and it brings the agentic loop into the editor rather than the terminal. KiloCode supports the same OpenAI-compatible provider configuration as the other tools, so connecting it to LiteLLM is a one-field change in its settings JSON. Its most distinctive feature relative to the terminal-based tools is direct access to the VS Code language server: it can read diagnostics, navigate symbols, and trigger refactors through the editor's own intelligence layer rather than relying purely on the LLM's understanding of the codebase. The `kilocode/home/` directory in the workspace layout holds its global configuration and extension state, which I bind-mount into the VS Code server container when running Code Server as a containerized IDE.
 
**pi** (pi.dev, Node.js) takes a deliberately minimal stance, omitting built-in MCP, plan mode, and permission gates in favor of a package-based extensibility model. It supports OpenRouter directly through a provider block in `models.json`, making OpenRouter's free-tier models, including Gemini 2.5 Pro, DeepSeek Chat v3.5, and LLaMA 4 Maverick, immediately available without any intermediate gateway. NVIDIA NIM endpoints are equally accessible through the same provider mechanism, since pi's model configuration accepts any OpenAI-compatible base URL. I reach for pi primarily for rapid exploratory work, precisely because it does not impose an opinionated workflow, and the combination of OpenRouter free-tier routing and optional NIM endpoints means the effective model selection available to it is substantially larger than the local Ollama catalog alone.
 
**Hermes** is not a third-party tool in the conventional sense but rather a named agent identity I maintain around Nous Research's Hermes 3 model family. The `hermes/home/` directory holds a configuration and prompt library tuned for Hermes 3's specific instruction format, function-calling conventions, and system prompt structure. Because Hermes 3 handles structured output and tool-use with uncommon consistency, I route all tool-calling-heavy workloads to this identity rather than to the generic LiteLLM endpoint. The identity acts as a specialization layer: the same model weight accessed through the Hermes identity behaves more reliably for agentic tasks than when accessed through a generic chat interface, because the system prompt and few-shot examples in the identity's configuration are calibrated to that model's particular strengths.
 
**GNHF** (Good Night Have Fun) is the task-bounded harness I use when I want a single-purpose agent that executes a defined workflow, reports results, and stops, without the persistent session management or self-modification behavior of the full agent frameworks. The name is a ham radio sign-off, which fits its character: it is polite, brief, and does exactly what it was asked to do. GNHF is structured as an npm package with its own home, config, and cache directories, and it runs inside a dedicated Docker container with a workspace mount scoped to the task at hand. Unlike Claude Code or pi, which are interactive and session-oriented, GNHF takes a task description and a workspace path as inputs, executes against the LiteLLM gateway, writes its outputs to the workspace, and exits. I use it for batch processing workflows, periodic maintenance tasks (linting a knowledge base, summarizing a document set, generating a report from structured data), and any scenario where I want an agent to run unattended without risk of a runaway interactive session accumulating unexpected state.
 
```bash
docker run --rm \
  --add-host=host.docker.internal:host-gateway \
  -e LITELLM_BASE_URL=http://host.docker.internal:4000/v1 \
  -e LITELLM_API_KEY=sk-litellm-local \
  -v "$HOME/agents/gnhf/home:/root/.gnhf" \
  -v "$HOME/projects/current:/workspace:ro" \
  -v "$HOME/agents/skills/core:/skills:ro" \
  my-gnhf:latest \
  gnhf run --task "summarize all markdown files in /workspace/raw" \
           --output /workspace/wiki/summaries
```
 
All six tools use project context files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) to provide persistent project instructions without consuming prompt tokens on every turn, and all six are converging on MCP as the standard for tool integration.
 
## The Google Ecosystem: Agents CLI and Workspace CLI
 
Two Google-specific tools occupy their own tier in the stack, with independent container identities and a shared philosophy of treating Google's API surface as a set of agent-accessible tools rather than endpoints for human-facing applications.
 
**Google Agents CLI** is the command-line interface for Google's Agent Development Kit (ADK), a Python framework for building multi-agent systems that run on Google's infrastructure and interact with Gemini models. The ADK CLI manages agent definitions, runs evaluation harnesses against defined test sets, and handles deployment to Vertex AI agent endpoints. Its workspace layout, `googleagentscli/data/`, `config/`, `cache/uv/`, `cache/npm/`, `evals/`, and `logs/`, reflects the breadth of what it manages: the `uv` cache indicates a Python-heavy dependency footprint, the `evals/` directory holds evaluation datasets and result logs for agent behavior testing, and the `logs/` directory accumulates structured run logs that I use to track agent performance across sessions. I run the ADK CLI in a dedicated container to isolate its Python environment from the host and from other Python-based tools in the stack.
 
```bash
docker run --rm -it \
  -v "$HOME/agents/googleagentscli/data:/app/data" \
  -v "$HOME/agents/googleagentscli/config:/root/.config/google-adk" \
  -v "$HOME/agents/googleagentscli/cache/uv:/root/.cache/uv" \
  -v "$HOME/agents/googleagentscli/evals:/app/evals" \
  -v "$HOME/agents/workspace:/workspace" \
  -e GOOGLE_API_KEY="$GOOGLE_API_KEY" \
  my-google-adk:latest
```
 
The primary use case I have for the ADK CLI is evaluation: I use its built-in harness to run structured tests against agent configurations before promoting them to production use, which provides a reproducible quality gate that purely interactive testing cannot.
 
**Google Workspace CLI** is a containerized `gcloud` environment configured with the scopes necessary to drive the Google Workspace APIs programmatically: Gmail, Drive, Calendar, Sheets, and Docs. The `googleworkspacecli/gcloud/` directory holds the application default credentials, active configuration, and any cached API tokens. The practical function this serves is enabling agents in the stack to read from and write to Workspace as a data layer: a document summarization task can pull source PDFs from Drive, a scheduling agent can query Calendar for availability, and a reporting workflow can write structured output directly to a Sheet. Rather than building custom integrations for each of these surfaces, the containerized Workspace CLI exposes them as shell commands that any agent with a bash tool can invoke.
 
```bash
# Inside an agent container with the workspace CLI mounted
gcloud auth application-default print-access-token
gcloud alpha storage cp gs://my-bucket/document.pdf /workspace/raw/
 
# List recent Drive files matching a query
gcloud alpha drive files list \
  --query="modifiedTime > '2026-04-01'" \
  --fields="files(id,name,mimeType)"
```
 
The authentication separation between the Workspace CLI container and the rest of the stack is intentional: the container that holds the Google credentials never has access to the project workspace or to any other agent's identity directory. Data flows through the workspace volume only.
 
## Agent Frameworks: Agent Zero, Archon, and Mastra
 
The stack runs three agent frameworks that occupy distinct positions on the spectrum from fully autonomous to fully programmable.
 
### Agent Zero
 
[Agent Zero](https://github.com/frdel/agent-zero) is the most autonomous framework in the stack, designed around the premise that the agent should be able to self-improve its own instructions and tools over the course of a session. It runs as a web UI on port 8081 and exposes a chat interface backed by a hierarchical agent system where the primary agent can spawn specialized subagents to handle subtasks, coordinate their outputs, and synthesize a result. The persistent state in `a0/data/` includes the agent's memory bank, its accumulated tool library, and its evolving system prompt, all of which carry forward across container restarts.
 
Agent Zero's Docker configuration uses a dedicated sandbox container for code execution, which provides a meaningful isolation boundary: tool calls that generate and run code happen inside the sandbox, not in the primary agent container or on the host. This is the design decision that makes it appropriate for exploratory and research workloads where the agent may generate and execute code it has not been explicitly instructed to run.
 
```bash
docker run -d \
  --name agent-zero \
  --add-host=host.docker.internal:host-gateway \
  -p 8081:80 \
  -v "$HOME/agents/a0/data:/a0/work_dir" \
  -e API_KEY="$LITELLM_API_KEY" \
  -e API_BASE="http://host.docker.internal:4000/v1" \
  frdel/agent-zero-run:latest
```
 
I use Agent Zero for research tasks that benefit from open-ended exploration: literature synthesis, multi-step data analysis where the analysis plan is not fully specified in advance, and any workflow where I want the agent to decide how to decompose a problem rather than following a predefined structure.
 
### Archon
 
[Archon](https://github.com/coleam00/Archon) occupies a meta-level in the stack: it is an agent framework whose purpose is to help build other agent frameworks. Its Streamlit-based UI presents a development environment where I describe the agent I want to build in natural language, and Archon generates the scaffolding, tool definitions, system prompt, and evaluation harness for that agent. The output is a structured agent project that can be deployed through any of the other framework layers in the stack.
 
Archon's state in `archon/data/` accumulates a library of previously generated agent templates and the associated evaluation results, which over time functions as a pattern library for agent design. When I need a new specialized agent, I start in Archon rather than from scratch, since the existing templates encode design decisions that have already been validated against the LiteLLM backend.
 
The practical integration between Archon and the rest of the stack is straightforward: Archon-generated agents are configured at generation time to use the LiteLLM endpoint, so they enter the stack already wired to the unified gateway without any post-generation modification.
 
### Mastra
 
[Mastra](https://mastra.ai) is a TypeScript-based AI agent framework that runs as a Docker Compose service exposing a REST API and a Studio UI on port 4111. It uses LibSQL for persistent conversation history, which means that agent memory survives container restarts. Mastra occupies the programmable end of the spectrum: rather than autonomous self-direction, it provides a typed API for defining agents, tools, workflows, and memory retrievers in TypeScript, with first-class support for multi-step workflows where each step's output is the next step's input.
 
The agent definition is deliberately minimal at the base level:
 
```typescript
import { Agent } from "@mastra/core/agent";
 
export const assistant = new Agent({
  name: "assistant",
  instructions: "You are a helpful, concise, and accurate assistant.",
  model: "openrouter/meta-llama/llama-3.1-8b-instruct:free",
});
```
 
The model string can be swapped without a rebuild by editing `assistant.ts` and running `docker compose up --build -d`. The agent server exposes a standard REST endpoint for programmatic integration with other tools in the stack:
 
```bash
curl -X POST http://localhost:4111/api/agents/assistant/generate \
  -H "Content-Type: application/json" \
  -d '{"messages": [{"role": "user", "content": "Summarize this document."}]}'
```
 
I use Mastra primarily as the API layer for agents that other services in the stack call programmatically, rather than as a user-facing interface. Its typed workflow system makes it well-suited for multi-step document processing pipelines, where each stage, extraction, classification, synthesis, and formatting, is a separately defined tool invocation with explicit input and output types.
 
## Docker Volume Architecture: Identity, Workspace, Skills
 
One of the more carefully considered design decisions in this stack is the separation of Docker bind mounts into four independent tiers, which I call identity, workspace, skills, and tool data. This separation means that swapping a user identity, adding a skills package, or destroying an experimental container affects only its own tier; the other three are untouched.
 
The identity hot-swap pattern is simple enough to describe in three commands: stop the container, remove it (volumes are untouched), and rerun with a different `home/` path. The workspace and skills mount points are identical in both invocations. This makes it straightforward to work on the same project files under different API key contexts or with different tool configurations.
 
Permission repair across containers with differing internal UID/GID values is handled by a disposable Alpine container that I use as a one-liner:
 
```bash
docker run --rm \
  -v "$HOME/agents/some-tool/home:/mnt/target" \
  alpine \
  chown -R 1000:1000 /mnt/target
```
 
On native Linux, bind-mount permissions are more predictable than under WSL2 because there is no UID translation layer. When a container runs as the same numeric UID as the host user, files created inside the container are owned by the host user automatically, and no repair step is needed. The repair one-liner is still useful for containers that run as a fixed non-root UID that differs from the host user's UID.
 
## Container-Isolated Tool Invocation
 
One of the more practically useful habits I have developed with this stack is running agentic CLI tools inside dedicated Docker containers rather than installing them to my host user environment. The motivation is threefold: environment isolation, workspace scope control, and plugin sandboxing.
 
**Environment isolation** is the most straightforward benefit. A containerized invocation carries only the environment variables, npm global installs, and configuration files that were explicitly placed in its bind-mounted home directory. There is no risk of a tool's dependencies conflicting with host-level node version constraints, no global npm package pollution, and no credential leakage between projects that use different API keys. When I want a clean slate, I delete the container (not the volume) and rerun; the tool reinstalls cleanly against a fresh ephemeral layer while the persistent home directory remains intact.
 
**Workspace scope control** is where the bind-mount architecture pays its most direct dividend. Rather than giving a tool access to the entire home filesystem, I mount only the specific project directories I want it to operate on:
 
```bash
docker run --rm -it \
  --add-host=host.docker.internal:host-gateway \
  -e ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY" \
  -v "$HOME/agents/commercial/claude/home:/root/.claude" \
  -v "$HOME/projects/mentir-ai:/workspace/mentir-ai" \
  -v "$HOME/projects/persist:/workspace/persist" \
  -w /workspace/mentir-ai \
  my-claude-code:latest
```
 
From inside the container, the agent sees exactly two project directories and nothing else. It cannot traverse up to the home directory, cannot read credential files for other services, and cannot inadvertently modify files in unrelated projects. For work involving student data or grant-sensitive materials, this mount-scoping discipline is not optional; it is the architectural enforcement of the data minimization principle.
 
**Plugin sandboxing** addresses a problem that anyone who actively evaluates emerging agentic tools encounters regularly: the ecosystem produces new MCP servers, pi extensions, Claude Code integrations, and npm-distributed tool packages at a pace that makes it impractical to vet each one thoroughly before trying it. The container boundary eliminates this problem. I can install an untrusted package, register a new MCP server, or try an experimental integration inside an ephemeral container, observe its behavior against a scoped workspace mount, and discard the container entirely if I decide against it, with no residue on the host.
 
The two-stage evaluation pattern, scratch container with a dedicated identity directory followed by deliberate promotion to the production identity, is something the tiered bind-mount architecture makes nearly effortless. The same pattern applies equally to pi extensions, KiloCode plugins, GNHF task modules, and ADK agent templates generated by Archon.
 
## Open-Source Cowork Alternatives
 
Anthropic's Claude Cowork launch in January 2026, which positioned a desktop-resident autonomous agent capable of managing files, executing terminal commands, and navigating browsers as the primary product surface, triggered a vigorous open-source response. I track several of the resulting projects.
 
[OpenWork](https://github.com/different-ai/openwork) is the most actively developed, functioning as a control surface for agentic workflows with hot-reloadable skills, session management, and SSE event stream subscriptions. It is ejectable to OpenCode, which provides a meaningful portability guarantee, and its skills model, where capabilities are defined as hot-reloadable modules rather than baked into the agent binary, is the design I find most extensible. [Accomplish](https://accomplish.ai/) takes a "BYO-AI" stance, functioning as the orchestration layer (hands and eyes) while allowing model selection, supporting OpenAI, Anthropic, Google, and Ollama backends without lock-in. [Kuse Cowork](https://github.com/kuse-ai/kuse_cowork) implements the agent runtime in Rust with Docker-based sandboxing, a meaningful departure from the Electron-first approach of many competing tools. [OpenCoworkAI](https://github.com/opencowork-ai/opencowork) explicitly commits to VM/bwrap isolation and checkpoint-rollback capability.
 
[Multica](https://github.com/multica-ai/multica) occupies a different conceptual position as a managed agents platform rather than a desktop agent. It assigns issues to AI agents as you would assign them to human teammates, and it implements skill compounding: when an agent completes a task successfully, the solution is saved as a reusable skill that future tasks can leverage. This maps interestingly onto organizational learning theory, though a thoughtful critique in the project's issue tracker notes that the human-management metaphor may be insufficient for genuinely autonomous AI orchestration at scale. I find this a productive tension worth thinking about seriously.
 
## MCP Servers: Extending the Stack with Custom Tools
 
The Model Context Protocol (MCP), introduced by Anthropic in late 2024, defines how AI assistants communicate with external tools through four primitive types: tools (callable functions), resources (readable data streams), prompts (reusable templates), and sampling (delegated inference requests). Running MCP servers in Docker and connecting them to the local stack over a shared external network is straightforward.
 
```bash
docker network create mcp-shared
```
 
With this network in place, any container that declares `mcp-shared` as an external network can reach an MCP server at its container DNS name, such as `http://mcp-bibliography:8000/mcp`, without host port exposure. Open WebUI version 0.4 and later natively enumerates and calls MCP tools when a function-calling capable model detects a tool-call response. Models I use for Open WebUI tool-calling sessions include `hermes3:8b`, `llama3.1:8b`, `qwen2.5:7b`, `qwen2.5:14b`, and `mistral-nemo:12b`.
 
For backends that use OpenAI function-calling format rather than MCP's JSON-RPC protocol, a thin Flask adapter service translates between the two on startup, enumerating available tools via `tools/list` and proxying function-call requests to `tools/call`:
 
```python
def mcp_post(method: str, params: dict, req_id: int = 1) -> dict:
    payload = json.dumps({
        "jsonrpc": "2.0", "id": req_id,
        "method": method, "params": params
    }).encode()
    req = urllib.request.Request(
        MCP_URL, data=payload,
        headers={"Content-Type": "application/json"}, method="POST"
    )
    with urllib.request.urlopen(req, timeout=10) as r:
        return json.loads(r.read())
```
 
## OpenRouter: A Cloud Model Interface
 
[OpenRouter](https://openrouter.ai) serves as the cloud model interface for tasks I route away from local inference. It exposes multiple model providers through an OpenAI-compatible API surface, so the same client code that talks to the local LiteLLM gateway can talk to OpenRouter without modification. Model identifiers follow a provider-qualified format:
 
```
anthropic/claude-3-opus
openai/gpt-4o
mistralai/mixtral-8x7b
meta-llama/llama-3-70b-instruct
```
 
The free-tier model list changes, but as of early 2026 includes Gemini 2.5 Pro, DeepSeek Chat v3.5, and LLaMA 4 Maverick. I use OpenRouter as a fallback in the Mastra agent server, as the primary cloud provider for pi, and as the escalation path from GNHF when a batch task exceeds local model capability. All API key management is handled through environment variables; no key is ever written into a container image.
 
## Reproducibility: The One-Shot Deploy Script
 
The entire stack, including all Dockerfiles, helper scripts, LiteLLM configuration, Mastra project files, and the directory tree, is generated by a single bash script called `deploy-agents.sh`. Running this script on a new machine, after providing the necessary API keys, produces a fully functional environment covering all the services described above with no manual steps.
 
The script follows a fixed sequence: create the directory tree, write Dockerfiles for each custom image, write LiteLLM and LocalAI configs, write Mastra project files, write GNHF task modules, write Agent Zero and Archon startup scripts, build custom images, pull pre-built images, and finally start all daemon containers. This design means the script itself is the documentation, and any configuration drift between machines is detected by diffing the generated files against a known-good reference.
 
Publishing a custom Docker image for any of these tools to GitHub Container Registry follows the same minimal GitHub Actions pattern:
 
```yaml
- uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}
 
- uses: docker/build-push-action@v6
  with:
    context: .
    push: true
    tags: ${{ steps.meta.outputs.tags }}
```
 
After the workflow runs, making the resulting package public in the GitHub package settings allows anyone to `docker pull ghcr.io/yourusername/yourrepo:main` without further configuration.
 
## Model Selection Notes
 
A few observations on local model selection from operational experience. For the think-heavy routing slot I use `gemma4:e2b`; for lightweight background tasks such as file summarization and classification I use `qwen2.5:1.5b`; for tool-calling workflows in Open WebUI and via the Hermes agent identity I use `hermes3:8b`; for general interactive sessions I use `llama3`. The selection criteria are principally RAM footprint and whether the model's function-calling format is well-supported by the tool in question.
 
The Arcee AI Trinity model family, released in April 2026, merits attention for anyone running hardware that can support 13 billion active parameters. Trinity Large uses a Mixture-of-Experts architecture with only 1.56% of its 400B total parameters active at inference time, achieving throughput roughly 2-3 times faster than dense models of equivalent parameter count. The Trinity-Large-Thinking reasoning variant is released under Apache 2.0 and is priced at approximately $0.90 per million output tokens on the Arcee API, placing it within reach for research and development use cases where local inference is insufficient. I intend to evaluate it against the Hermes identity configuration, since reasoning variants that also maintain strong function-calling fidelity are genuinely rare.
