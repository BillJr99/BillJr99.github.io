For the past several years I have been thinking carefully about what it means to run AI infrastructure that I actually own, control, and understand from the ground up. The rapid proliferation of frontier model APIs, agentic coding tools, and open-weight model releases in 2025-2026 finally made this tractable at a price and complexity point that a single person could manage. This post documents the architecture I settled on: a self-hosted, Docker-based stack running on a mini PC, unified by a single OpenAI-compatible model gateway, and surfaced through a collection of agentic CLI tools, open-source Cowork alternatives, and a command interface built around structured task queues. My goals were privacy, sovereignty, reproducibility, and the ability to swap components without rebuilding everything from scratch.

## Motivation: Why Self-Host?
 
The short answer is control. When I work on grant-funded AI research, on student data in the context of my courses at Ursinus, or on institutional planning through my role as Co-Director of the Teaching and Learning Institute, I want to know exactly where inference is happening and what data is leaving my environment. The longer answer is pedagogical: I cannot credibly teach AI literacy, AI ethics, and responsible deployment if I have not made serious, hands-on architectural decisions myself. Running your own stack is humbling in the right ways.
 
There is also a strategic argument. As the open-source AI agent framework ecosystem matured through early 2026, it became clear that the layered architecture of these systems, separating the model layer, the orchestration layer, the tool-access layer, and the user-interface layer, was stabilizing into recognizable patterns. A well-designed self-hosted stack can plug into any of these layers without being locked to a single vendor. That flexibility is worth the setup cost.
 
## Hardware: The Mini PC
 
The physical foundation of the stack is a mini PC running Linux Mint, which gives me a clean Debian-lineage environment with full access to the upstream Docker Engine repository and no virtualization layer between the container workloads and the host kernel. Everything runs directly on the host, which simplifies networking, volume permissions, and service lifecycle management considerably compared to hypervisor-based setups.
 
### Installing Docker Engine
 
Linux Mint ships an older Docker build from the distribution mirror, so I install from the upstream Docker repository instead. The setup sequence is:
 
```bash
# Remove any distribution-packaged versions
sudo apt-get remove docker docker-engine docker.io containerd runc
 
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release
 
# Add Docker's official GPG key
sudo mkdir -m 0755 -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
 sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
 
# Point the apt source at the Ubuntu codename that Mint is based on
echo \
 "deb [arch=$(dpkg --print-architecture) \
 signed-by=/etc/apt/keyrings/docker.gpg] \
 https://download.docker.com/linux/ubuntu \
 $(. /etc/os-release && echo $UBUNTU_CODENAME) stable" | \
 sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
 
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io \
 docker-buildx-plugin docker-compose-plugin
 
# Verify
sudo docker run hello-world
 
# Allow your user to run Docker without sudo
sudo usermod -aG docker $USER
newgrp docker
```
 
The `UBUNTU_CODENAME` variable in the source line is the key detail for Mint: it resolves to the underlying Ubuntu release name rather than the Mint release name, which is what the Docker repository actually indexes.
 
### Installing Ollama
 
Ollama runs as a native Linux service, not inside a container, which keeps model-weight I/O off the Docker networking path and avoids bind-mount overhead for the weight files:
 
```bash
curl -fsSL https://ollama.com/install.sh | sh
# The installer registers a systemd service; verify it started
systemctl status ollama
```
 
Ollama listens on `http://localhost:11434` by default and is reachable from within containers via `--add-host=host.docker.internal:host-gateway`, which every container in the stack declares. This flag is required on Linux Docker Engine; unlike Docker Desktop, the Linux engine does not inject `host.docker.internal` automatically.
 
### Workspace Layout
 
All agent state lives under a single root in the home directory, which keeps it easy to back up, version, and hand off to another machine:
 
```
~/agents/
├── workspace/          # Shared project files (all TUI tools mount here)
├── skills/core/        # Read-only skills package (mounted :ro)
├── litellm/            # LiteLLM config and Compose files
├── kilocode/home/      # KiloCode identity
├── openclaude/home/    # OpenClaude identity (~/.claude inside container)
├── opencode/home/      # opencode.ai identity
├── pi/home/            # pi.dev identity, models.json, settings.json
├── commercial/         # Claude Code, Codex, Gemini CLI, CCR, Copilot
│   ├── claude/         # Per-tool: workspace/, home/, npm/, config/, cache/
│   ├── codex/
│   ├── gemini/
```

Creating the full tree is a one-liner:

On native Linux there is no drive-letter isolation, no WSL boundary to cross, and no `automount` or `appendWindowsPath` configuration to manage. The workspace is simply a directory in the home filesystem, accessible to Docker via ordinary bind mounts and to the host user with full POSIX permissions.
 
## The Core Design Principle: A Unified Model Gateway
 
The single most important architectural decision I made was to route all LLM inference through a single OpenAI-compatible endpoint rather than having each tool reach out to Ollama, Anthropic, or OpenRouter independently. I use [LiteLLM](https://github.com/BerriAI/litellm) for this. Every service in the stack sends its requests to `http://localhost:4000/v1` with the bearer token `sk-litellm-local`. LiteLLM translates those requests to whatever backend is appropriate, whether a local Ollama model or an OpenRouter free-tier endpoint, according to a YAML routing configuration.
 
```yaml
model_list:
 - model_name: llama3
   litellm_params:
     model: ollama/llama3
     api_base: http://host.docker.internal:11434
 
 - model_name: qwen2.5-3b
   litellm_params:
     model: ollama/qwen2.5:3b
     api_base: http://host.docker.internal:11434
 
 - model_name: openrouter/auto
   litellm_params:
     model: openrouter/auto
     api_key: ${OPENROUTER_API_KEY}
     api_base: https://openrouter.ai/api/v1
```
 
The practical consequence is that changing a model, adding a provider, or adjusting routing requires editing one YAML file and restarting one service. No other container needs to know that anything changed. This is the same composability principle I teach in software engineering: minimize coupling, maximize cohesion.
 
Connecting a commercial CLI tool like Claude Code to the local stack requires only two environment variable exports:
 
```bash
export ANTHROPIC_BASE_URL=http://localhost:4000
export ANTHROPIC_API_KEY=sk-litellm-local
```
 
From Claude Code's perspective, nothing has changed. From the infrastructure's perspective, every token has been processed locally.
 
## The Full Stack: Services and Ports
 
Running everything in Docker Compose or as standalone containers gives me clean lifecycle management, easy restarts, and clear separation between persistent identity state (credentials, config) and ephemeral compute. The full service map is as follows.
 
| Service | Port | Purpose |
|---------|------|---------|
| LiteLLM | 4000 | Unified model gateway (OpenAI-compatible) |
| Ollama | 11434 | Local LLM inference (native systemd service) |
| Open WebUI | 3000 | Browser-based LLM frontend |
| Mastra | 4111 | TypeScript AI agent server (API + Studio UI) |
| LocalAI | 8080 | GGUF model inference (OpenAI-compatible) |
| Agent Zero | 8081 | Autonomous web-based agent UI |
| Openwork | 8787 | Agentic orchestrator UI |
| Portainer | 9000 | Docker management UI |
 
The local models I pull via Ollama, configured through LiteLLM's alias table, include `phi4-mini`, `smollm2`, `gemma4:e2b`, `qwen2.5:1.5b`, `qwen2.5:3b`, and `llama3`. The model selection is intentionally conservative by RAM footprint: these models run comfortably without GPU acceleration and without swapping. For complex drafting, long-context analysis, or anything requiring stronger reasoning, LiteLLM escalates to OpenRouter's free tier transparently.
 
## Docker Volume Architecture: Identity, Workspace, Skills
 
One of the more carefully considered design decisions in this stack is the separation of Docker bind mounts into four independent tiers, which I call identity, workspace, skills, and tool data. This separation means that swapping a user identity, adding a skills package, or destroying an experimental container affects only its own tier; the other three are untouched.
 
The identity hot-swap pattern is simple enough to describe in three commands: stop the container, remove it (volumes are untouched), and rerun with a different `home/` path. The workspace and skills mount points are identical in both invocations. This makes it straightforward to work on the same project files under different API key contexts or with different tool configurations.
 
Permission repair across containers with differing internal UID/GID values is handled by a disposable Alpine container that I use as a one-liner:
 
```bash
docker run --rm \
 -v "$HOME/agents/openclaude/home:/mnt/target" \
 alpine \
 chown -R 1000:1000 /mnt/target
```
 
On native Linux, bind-mount permissions are more predictable than under WSL2 because there is no UID translation layer. When a container runs as the same numeric UID as the host user, files created inside the container are owned by the host user automatically, and no repair step is needed. The repair one-liner is still useful for containers that run as a fixed non-root UID that differs from the host user's UID.
 
## Agentic CLI Tools: A Comparative Survey
 
By April 2026, five mature agentic coding CLI tools have emerged with distinct architectural philosophies, and I run all of them through the unified LiteLLM gateway. The comparative picture is useful to have in one place.
 
**Claude Code** (Anthropic, Node.js) is the most fully featured in terms of built-in subagent support, MCP integration, and permission gate granularity. It uses `CLAUDE.md` files for project context and `.claude/agents/` Markdown files for custom subagent definitions. Connecting it to the local stack requires only setting `ANTHROPIC_BASE_URL` and `ANTHROPIC_API_KEY`.
 
**OpenAI Codex CLI** (Rust) is the most efficient in terms of resource consumption and supports native multi-provider configuration through a TOML config file. Custom providers are defined as named sections:
 
```toml
model = "llama3.3:70b"
model_provider = "openwebui"
 
[model_providers.openwebui]
name = "Open WebUI"
base_url = "http://localhost:3000/openai"
env_key = "OPENWEBUI_API_KEY"
```
 
**Gemini CLI** (Google, Node.js) uses `GEMINI.md` files for project context and a three-tier discovery hierarchy for skills. Routing it through an OpenAI-compatible endpoint requires the `open-gemini-cli` fork, which injects an adapter layer that translates Gemini's internal message format.
 
**OpenCode** (opencode.ai, Go) is the most flexible in terms of provider support, relying on the `@ai-sdk/openai-compatible` adapter to connect to any OpenAI-compatible backend, including the local LiteLLM instance:
 
```json
{
 "provider": {
   "litellm": {
     "npm": "@ai-sdk/openai-compatible",
     "options": { "baseURL": "http://localhost:4000/v1" },
     "models": { "llama3": {}, "qwen2.5-3b": {} }
   }
 }
}
```
 
**pi** (pi.dev, Node.js) takes a deliberately minimal stance, omitting built-in MCP, plan mode, and permission gates in favor of a package-based extensibility model. It supports OpenRouter directly and integrates with local Ollama through a custom provider block in `models.json`. The pi agent is what I find most useful for rapid exploratory work precisely because it does not impose an opinionated workflow.
 
All five tools use project context files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) to provide persistent project instructions without consuming prompt tokens on every turn, and all five are converging on MCP as the standard for tool integration, with LangGraph and AutoGen having the most mature implementations as of early 2026.
 
## Claude Code Router: Intelligent Local Routing
 
One of the more practically useful components in the commercial tool container is Claude Code Router (CCR), which sits between the `claude` CLI and the model backends, routing requests to different models based on the nature of the task. The routing configuration is a JSON file that maps slot names to model identifiers:
 
```json
{
 "Router": {
   "default": "ollama,llama3:latest",
   "background": "ollama,qwen2.5:1.5b",
   "think": "ollama,gemma4:e2b",
   "longContext": "openrouter,google/gemini-2.5-pro-exp-03-25:free",
   "longContextThreshold": 60000,
   "webSearch": "openrouter,google/gemini-2.5-pro-exp-03-25:online"
 }
}
```
 
The practical effect is that routine code edits, file summarization, and inline completions go to a small local model with sub-second latency, while tasks requiring extended reasoning or more than 60,000 tokens of context are escalated to OpenRouter's free Gemini tier. Switching the active model within a running Claude Code session is a single slash command: `/model ollama,llama3:latest`.
 
## Open-Source Cowork Alternatives
 
Anthropic's Claude Cowork launch in January 2026, which positioned a desktop-resident autonomous agent capable of managing files, executing terminal commands, and navigating browsers as the primary product surface, triggered a vigorous open-source response. I track several of the resulting projects.
 
[OpenWork](https://github.com/different-ai/openwork) is the most actively developed, functioning as a control surface for agentic workflows with hot-reloadable skills, session management, and SSE event stream subscriptions. It is ejectable to OpenCode, which provides a meaningful portability guarantee. [Accomplish](https://accomplish.ai/) takes a "BYO-AI" stance, functioning as the orchestration layer (hands and eyes) while allowing model selection, supporting OpenAI, Anthropic, Google, and Ollama backends without lock-in. [Kuse Cowork](https://github.com/kuse-ai/kuse_cowork) implements the agent runtime in Rust with Docker-based sandboxing, a meaningful departure from the Electron-first approach of many competing tools. [OpenCoworkAI](https://github.com/opencowork-ai/opencowork) explicitly commits to VM/bwrap isolation and checkpoint-rollback capability.
 
[Multica](https://github.com/multica-ai/multica) occupies a different conceptual position as a managed agents platform rather than a desktop agent. It assigns issues to AI agents as you would assign them to human teammates, and it implements skill compounding: when an agent completes a task successfully, the solution is saved as a reusable skill that future tasks can leverage. This maps interestingly onto organizational learning theory, though a thoughtful critique in the project's issue tracker notes that the human-management metaphor may be insufficient for genuinely autonomous AI orchestration at scale. I find this a productive tension worth thinking about seriously.
 
## MCP Servers: Extending the Stack with Custom Tools
 
The Model Context Protocol (MCP), introduced by Anthropic in late 2024, defines how AI assistants communicate with external tools through four primitive types: tools (callable functions), resources (readable data streams), prompts (reusable templates), and sampling (delegated inference requests). Running MCP servers in Docker and connecting them to the local stack over a shared external network is straightforward.
 
```bash
docker network create mcp-shared
```
 
With this network in place, any container that declares `mcp-shared` as an external network can reach an MCP server at its container DNS name, such as `http://mcp-bibliography:8000/mcp`, without host port exposure. Open WebUI version 0.4 and later natively enumerates and calls MCP tools when a function-calling capable model detects a tool-call response. Models confirmed to work reliably with Open WebUI tool calling include `llama3.1:8b`, `qwen2.5:7b`, `qwen2.5:14b`, `mistral-nemo:12b`, and `hermes3:8b`.
 
For backends that use OpenAI function-calling format rather than MCP's JSON-RPC protocol, a thin Flask adapter service translates between the two on startup, enumerating available tools via `tools/list` and proxying function-call requests to `tools/call`:
 
```python
def mcp_post(method: str, params: dict, req_id: int = 1) -> dict:
   payload = json.dumps({
       "jsonrpc": "2.0", "id": req_id,
       "method": method, "params": params
   }).encode()
   req = urllib.request.Request(
       MCP_URL, data=payload,
       headers={"Content-Type": "application/json"}, method="POST"
   )
   with urllib.request.urlopen(req, timeout=10) as r:
       return json.loads(r.read())
```
 
## OpenRouter: Free-Tier Cloud Escalation
 
[OpenRouter](https://openrouter.ai) functions as the cloud escalation layer for tasks that exceed local model capability. It exposes multiple model providers through an OpenAI-compatible API surface, so the same client code that talks to the local LiteLLM gateway can talk to OpenRouter without modification. Model identifiers follow a provider-qualified format:
 
```
anthropic/claude-3-opus
openai/gpt-4o
mistralai/mixtral-8x7b
meta-llama/llama-3-70b-instruct
```
 
The free-tier model list changes, but as of early 2026 includes Gemini 2.5 Pro, DeepSeek Chat v3.5, and LLaMA 4 Maverick. I use OpenRouter for the long-context and web-search routing slots in CCR, and as a fallback in the Mastra agent server for tasks where local models are insufficient. All API key management is handled through environment variables; no key is ever written into a container image.
 
## The Mastra Agent Server
 
[Mastra](https://mastra.ai) is a TypeScript-based AI agent framework that runs as a Docker Compose service exposing a REST API and a Studio UI on port 4111. It uses LibSQL for persistent conversation history, which means that agent memory survives container restarts. The agent definition is deliberately minimal:
 
```typescript
import { Agent } from "@mastra/core/agent";
 
export const assistant = new Agent({
 name: "assistant",
 instructions:
   "You are a helpful, concise, and accurate assistant.",
 model: "openrouter/meta-llama/llama-3.1-8b-instruct:free",
});
```
 
The model string can be swapped without a rebuild by editing `assistant.ts` and running `docker compose up --build -d`. The agent server exposes a standard REST endpoint for programmatic integration with other tools in the stack:
 
```bash
curl -X POST http://localhost:4111/api/agents/assistant/generate \
 -H "Content-Type: application/json" \
 -d '{"messages": [{"role": "user", "content": "Summarize this document."}]}'
```
 
## Reproducibility: The One-Shot Deploy Script
 
The entire stack, including all Dockerfiles, helper scripts, LiteLLM configuration, Mastra project files, CCR routing config, and directory tree, is generated by a single bash script called `deploy-agents.sh`. Running this script on a new machine, after providing an OpenRouter API key, produces a fully functional environment with no manual steps beyond placing a Dockerfile in the `gnhf/` directory for the task-bounded agent harness.
 
The script follows a nine-step sequence: create the directory tree, write Dockerfiles, write LiteLLM configs, write Mastra project files, write gnhf scripts, write Agent Zero scripts, build custom images, clone and build OpenClaude, pull pre-built images, and finally start all daemon containers. This design means the script itself is the documentation, and any configuration drift between machines is detected by diffing the generated files.
 
Publishing a custom Docker image for any of these tools to GitHub Container Registry follows the same minimal GitHub Actions pattern:
 
```yaml
- uses: docker/login-action@v3
 with:
   registry: ghcr.io
   username: ${{ github.actor }}
   password: ${{ secrets.GITHUB_TOKEN }}
 
- uses: docker/build-push-action@v6
 with:
   context: .
   push: true
   tags: ${{ steps.meta.outputs.tags }}
```
 
After the workflow runs, making the resulting package public in the GitHub package settings allows anyone to `docker pull ghcr.io/yourusername/yourrepo:main` without further configuration.
 
## Model Selection Notes
 
A few observations on local model selection that are worth sharing from operational experience. The `gemma4:e2b` checkpoint is a good choice for the "think" routing slot in CCR because it handles multi-step reasoning better than its parameter count suggests, at a RAM footprint that fits comfortably alongside other services. The `qwen2.5:1.5b` model is the lowest-latency local option and is appropriate for the background slot, handling file summarization, housekeeping, and lightweight classification tasks. `llama3` remains the most reliable choice for tool-calling workflows in Open WebUI because its instruction following is consistent and its function-calling format is well-supported across the ecosystem.
 
The Arcee AI Trinity model family, released in April 2026, merits attention for anyone running hardware that can support 13 billion active parameters. Trinity Large uses a Mixture-of-Experts architecture with only 1.56% of its 400B total parameters active at inference time, achieving throughput roughly 2-3 times faster than dense models of equivalent parameter count. The Trinity-Large-Thinking reasoning variant is released under Apache 2.0 and is priced at approximately $0.90 per million output tokens on the Arcee API, placing it within reach for research and development use cases where local inference is insufficient.
